Tổng quan

1 WAF WebACL với 6 rule groups bảo vệ Public ALB — gated trước EC2 webapp.

▾
🔴

⚠️ ĐÂY LÀ PRODUCTION — WAF đang bảo vệ BankSystem, CRM, E-Approval, FeedProd. Thay đổi rule sai có thể block legitimate users hoặc tắt protection.

Định nghĩa

AWS WAF v2 là firewall Layer 7 — inspect HTTP/HTTPS request và block/allow/count dựa theo rules. Japfa dùng WebACL gắn vào Public ALB để chặn các attack phổ biến trước khi traffic reach EC2 webapp IIS.

Cấu hình qua Console

Xem WAF WebACL, rule metrics và blocked requests qua Console.

▾
  1. Đăng nhập Console jp:prod — WAF & Shield

    SSO → jp:prod (347486023795) → Search: WAF & Shield → WAF → Web ACLs.

    Region phải là ap-southeast-1 (WAF REGIONAL cho ALB).

    Nếu chọn "Global" hoặc CloudFront scope sẽ không thấy WAF này — WAF Japfa là REGIONAL
  2. Xem Rule Groups và Metrics

    Click vào jp-prod-waf-aps1-webapp → tab Rules.

    Xem danh sách 6 rule groups theo priority. Click từng rule → xem CloudWatch metrics (AllowedRequests, BlockedRequests, CountedRequests).

    MetricÝ nghĩaCần lo khi nào
    BlockedRequestsSố request bị blockTăng đột biến = đang bị attack
    AllowedRequestsSố request được cho quaGiảm đột biến = WAF block quá mức (false positive)
    CountedRequests (Bot)Bot traffic đang trackCao = bot scraping
  3. Xem Sampled Requests (debug false positive)

    Tab Overview → Sampled requests → chọn rule group → xem sample requests bị block.

    Mỗi sample có: timestamp, URI, source IP, HTTP method, matched rule, action.

    Nếu thấy legitimate user bị block → xem request details để tạo exception rule cho pattern đó
  4. Xem Associated Resources

    Tab Associated AWS resources → xác nhận chỉ có 1 resource: Public ALB ARN.

    WAF KHÔNG được gắn vào API Gateway hay Internal ALB — verify đúng chỉ 1 resource.

Cấu hình qua CLI

AWS CLI để xem WAF WebACL, rules và metrics.

▾
ℹ

WAF v2 API dùng service wafv2, không phải waf (waf = cũ v1). Phải thêm --scope REGIONAL cho ALB-attached WAF.

AWS CLI · WAFv2 Read Commands🟢 READ ONLY · jp:prod
# Xem tất cả WebACLs (REGIONAL = cho ALB/API GW)
aws wafv2 list-web-acls \
  --scope REGIONAL \
  --profile jp:prod --region ap-southeast-1 \
  --query 'WebACLs[*].{Name:Name,ID:Id,ARN:ARN}' \
  --output table

# Xem chi tiết WebACL và rules (cần cả Name và ID)
aws wafv2 get-web-acl \
  --name jp-prod-waf-aps1-webapp \
  --scope REGIONAL \
  --id {{webacl-id}} \
  --profile jp:prod --region ap-southeast-1

# Xem resources đang được protect bởi WebACL
aws wafv2 list-resources-for-web-acl \
  --web-acl-arn {{webacl-arn}} \
  --profile jp:prod --region ap-southeast-1

# Xem sampled requests từ rule group (đang bị block gì)
aws wafv2 get-sampled-requests \
  --web-acl-arn {{webacl-arn}} \
  --rule-metric-name AWSManagedRulesCommonRuleSet \
  --scope REGIONAL \
  --time-window StartTime=$(date -u -v-1H +%Y-%m-%dT%H:%M:%SZ),EndTime=$(date -u +%Y-%m-%dT%H:%M:%SZ) \
  --max-items 100 \
  --profile jp:prod --region ap-southeast-1

# Xem managed rule groups có sẵn của AWS
aws wafv2 list-available-managed-rule-groups \
  --scope REGIONAL \
  --profile jp:prod --region ap-southeast-1 \
  --query 'ManagedRuleGroups[*].{Name:Name,Vendor:VendorName}' \
  --output table

→Thêm IP vào whitelist (🔴 Production — xác nhận 2 lần)

AWS CLI · WAFv2 IP Set (Whitelist)🔴 WRITE · jp:prod
# Tạo IP Set whitelist (ví dụ: cho phép IP office Japfa bypass WAF check)
aws wafv2 create-ip-set \
  --name jp-prod-waf-ipset-whitelist \
  --scope REGIONAL \
  --ip-address-version IPV4 \
  --addresses "{{IP/32}" \           # Chỉ ghi IP cần whitelist
  --profile jp:prod --region ap-southeast-1

# Cập nhật IP Set (thêm IP mới vào existing set)
aws wafv2 update-ip-set \
  --name jp-prod-waf-ipset-whitelist \
  --scope REGIONAL \
  --id {{ipset-id}} \
  --lock-token {{lock-token}} \    # Lấy từ get-ip-set
  --addresses "{{IP1/32}" "{{IP2/32}" \
  --profile jp:prod --region ap-southeast-1

Cấu hình qua Terraform

WAF WebACL trong TF jp:prod.

▾
Terraform · waf.tfHCL
resource "aws_wafv2_web_acl" "webapp" {
  name        = "jp-prod-waf-aps1-webapp"
  description = "WAF protection for Japfa webapp — OWASP, SQLi, Bot, Bad IP"
  scope       = "REGIONAL"             # REGIONAL cho ALB; CLOUDFRONT nếu gắn CloudFront

  default_action { allow {} }           # Default ALLOW — WAF chặn bad, không chặn tất cả

  # Rule 1: OWASP Common
  rule {
    name     = "AWSManagedRulesCommonRuleSet"
    priority = 1
    override_action { none {} }         # Dùng action của managed rule (Block/Count)
    statement {
      managed_rule_group_statement {
        vendor_name = "AWS"
        name        = "AWSManagedRulesCommonRuleSet"
      }
    }
    visibility_config {
      cloudwatch_metrics_enabled = true
      metric_name                = "AWSManagedRulesCommonRuleSet"
      sampled_requests_enabled   = true   # Bật để debug blocked requests
    }
  }

  # Rule 5: IP Reputation List
  rule {
    name     = "AWSManagedRulesAmazonIpReputationList"
    priority = 5
    override_action { none {} }
    statement {
      managed_rule_group_statement {
        vendor_name = "AWS"
        name        = "AWSManagedRulesAmazonIpReputationList"
      }
    }
    visibility_config {
      cloudwatch_metrics_enabled = true
      metric_name                = "AWSManagedRulesAmazonIpRepList"
      sampled_requests_enabled   = true
    }
  }
  # ... rules 2-4, 6 tương tự với priority và name tương ứng

  visibility_config {
    cloudwatch_metrics_enabled = true
    metric_name                = "jp-prod-waf-aps1-webapp"
    sampled_requests_enabled   = true
  }
  tags = { Name = "jp-prod-waf-aps1-webapp", Environment = "production" }
}

Ưu & Nhược điểm

WAF Managed Rules vs Custom Rules vs không có WAF.

▾

Ưu điểm

  • AWS Managed Rules — AWS tự cập nhật signature, không phải maintain rule thủ công
  • OWASP Top 10 coverage — 6 rule groups cover XSS, SQLi, LFI, bad inputs, bot, bad IP
  • Tích hợp ALB — Inspect request trước khi reach EC2 — EC2 không biết bad request
  • Sampled Requests — Debug được request nào bị block và lý do
  • Count mode — Test rule mới ở Count trước Block để tránh false positive

Nhược điểm / Rủi ro

  • False positive — Managed Rules đôi khi block legitimate request (IIS forms, đặc biệt SQLi rule với query có từ "select")
  • Chi phí tăng theo traffic — WCU-based pricing; 6 rule groups = cao hơn 1-2 rules
  • Không customize deep — Managed Rules là black box; không tune chi tiết như WAF custom rule
  • Log storage cost — WAF log gửi Kinesis Firehose → S3 nếu bật full logging

Checklist Thêm / Sửa / Xóa

Checklist WAF operations — sai có thể block production users.

▾

→Checklist: Thêm rule mới vào WAF

  • Thêm rule ở Count mode trước — Không thêm Block thẳng; Count 1-2 ngày để verify không false positive
  • Xem Sampled Requests sau 1 ngày — Kiểm tra request bị Count có phải legitimate không
  • Tạo exception nếu cần — Nếu có false positive: tạo IP Set hoặc regex exception trước khi đổi sang Block
  • Đổi sang Block — Sau khi verify Count mode không false positive → đổi action sang Block

→Checklist: Xử lý False Positive (user bị block nhầm)

  • Xem Sampled Requests — Tìm request của user đó: URI, Rule triggered, Action
  • Identify rule gây false positive — Rule nào trigger? (CommonRuleSet / SQLi / ...)
  • Option A: Whitelist IP user — Tạo IP Set + rule priority cao hơn với action Allow — nhanh nhất
  • Option B: Override rule sang Count — Trong managed rule group, override specific rule sang Count (không block toàn group)
  • Verify user không còn bị block — Test lại request từ user

→Checklist: Disassociate WAF (xóa protection) — NGUY HIỂM

  • Xác nhận 2 lần với 3T — Disassociate = EC2 webapp exposed internet không có WAF
  • Chỉ làm trong maintenance window — Tạm thời trong khi đổi WAF WebACL, không để lâu
  • Associate ngay WAF mới trước khi disassociate cũ — Minimize exposure window

Best Practices & Tài liệu

Nguyên tắc vận hành WAF Japfa.

▾

→Best Practices

Triển khai

  • Test Count mode trước Block — Mọi rule mới phải qua Count 1-2 ngày để phát hiện false positive
  • Bật sampled_requests_enabled — Không bật thì không debug được request nào bị block
  • Đặt priority hợp lý — IP whitelist phải có priority thấp hơn số (= ưu tiên cao hơn) so với block rules

Monitoring

  • CloudWatch alarm BlockedRequests spike — Tăng đột biến = đang bị attack; giảm đột biến = WAF bị bypass hoặc false negative
  • Monitor AllowedRequests không giảm bất thường — Giảm mạnh = WAF đang block over-aggressively

Security

  • WAF không thay thế SG — WAF là Layer 7; SG là Layer 4. Cần cả hai. WAF bypass = vẫn có SG bảo vệ port
  • Linux Rule Set có thể bỏ — Webapp là Windows IIS; Linux rules ít applicable nhưng không hại

→Gotchas

PITFALL

SQLi Rule block IIS form với từ "select" — AWSManagedRulesSQLiRuleSet hay false positive với form input chứa "select" trong dropdown HTML. Fix: override specific SQLi rule để Count hoặc tạo body regex exception.

PITFALL

Scope phải REGIONAL không phải CLOUDFRONT — WAF REGIONAL cho ALB. Nếu chọn CLOUDFRONT scope trong Console hoặc CLI sẽ không thấy WebACL này.

GOTCHA

WAF không attached API Gateway — Chỉ gắn Public ALB. API Gateway traffic không qua WAF Japfa hiện tại.

TIP

Dùng override_action none{} cho managed rule — TF: override_action { none {} } = giữ nguyên action của managed rule. override_action { count {} } = force Count toàn bộ rule group bất kể rule action.
