Cấu hình qua Console

Xem VPC, subnet, route table và RAM Share qua AWS Console (jp:network).

▾
⚠
Quan trọng: VPC/Subnet là P1 — thay đổi route table hoặc subnet config có thể làm toàn bộ hệ thống mất kết nối. Backup JSON trước mọi thay đổi.

→Xem VPC và Subnet

  1. Đăng nhập Console jp:network

    SSO → jp:network (787309524943) → Console → Region: ap-southeast-1.

    Search: VPC → Your VPCs → thấy 3 VPC của Japfa.

    Phải vào jp:network — nếu vào jp:prod chỉ thấy subnet được share, không thể sửa
  2. Xem Subnets và Route Tables

    VPC → Subnets → filter theo VPC jp-prod-vpc-aps1-production.

    Click từng subnet → tab Route table để xem route — quan trọng để verify airgap-db không có internet route.

    Subnet tierRoute bình thườngCần kiểm tra
    Public0.0.0.0/0 → IGWIGW đang attached và active
    Private-App0.0.0.0/0 → NAT GWNAT GW còn active, không bị xóa
    Airgap-DBKHÔNG có 0.0.0.0/0Chỉ có local route và TGW route — không có internet
    TGW-Attach0.0.0.0/0 → TGWTGW đang active
  3. Xem RAM Share status

    VPC → Resource Access Manager → Shared by me → tìm jp-prod-ram-aps1-prod-subnets.

    Xem tab Shared resources — tất cả 6 subnet phải có status Associated.

    Nếu RAM Share bị xóa hoặc subnet không Associated → jp:prod không launch EC2 được
  4. Xem VPC Flow Logs (nếu cần debug network)

    VPC → chọn VPC production → tab Flow Logs → xem logs đang được gửi về CloudWatch hoặc S3.

    Flow logs giúp debug: traffic bị reject, kết nối failed, SG blocking.

Cấu hình qua CLI

AWS CLI để xem VPC, subnet, route table từ jp:network.

▾
ℹ

Tất cả lệnh VPC dùng --profile jp:network — không phải jp:prod. jp:prod chỉ thấy shared resources, không thể modify.

→Lệnh đọc / kiểm tra (🟢 An toàn)

AWS CLI · Describe VPC & Subnets🟢 READ ONLY
# Xem tất cả VPC
aws ec2 describe-vpcs \
  --profile jp:network \
  --region ap-southeast-1 \
  --query 'Vpcs[*].{ID:VpcId,CIDR:CidrBlock,Name:Tags[?Key==`Name`]|[0].Value}' \
  --output table

# Xem tất cả subnets trong Production VPC
aws ec2 describe-subnets \
  --filters "Name=vpc-id,Values={{vpc-id-prod}}" \
  --query 'Subnets[*].{ID:SubnetId,CIDR:CidrBlock,AZ:AvailabilityZone,Name:Tags[?Key==`Name`]|[0].Value,Available:AvailableIpAddressCount}' \
  --profile jp:network --region ap-southeast-1 \
  --output table

# Xem route tables — verify airgap-db không có internet route
aws ec2 describe-route-tables \
  --filters "Name=vpc-id,Values={{vpc-id-prod}}" \
  --profile jp:network --region ap-southeast-1 \
  --output json

# Xem RAM Share đang share gì sang jp:prod
aws ram get-resource-shares \
  --resource-owner SELF \
  --profile jp:network --region ap-southeast-1 \
  --query 'resourceShares[*].{Name:name,Status:status,ARN:resourceShareArn}' \
  --output table

# Xem NAT Gateway status
aws ec2 describe-nat-gateways \
  --filter "Name=vpc-id,Values={{vpc-id-prod}}" \
  --query 'NatGateways[*].{ID:NatGatewayId,State:State,PublicIP:NatGatewayAddresses[0].PublicIp,Subnet:SubnetId}' \
  --profile jp:network --region ap-southeast-1 \
  --output table

→Thêm route vào route table (🔴 Nguy hiểm — ảnh hưởng toàn mạng)

🔴
CẢNH BÁO MẠNG: Sai route table có thể làm mất kết nối toàn bộ hạ tầng. Export JSON backup trước, xác nhận 2 lần với 3T.
AWS CLI · Modify Route Table🔴 WRITE — NETWORK IMPACT
# Backup route table hiện tại trước
aws ec2 describe-route-tables \
  --route-table-ids {{route-table-id}} \
  --profile jp:network --region ap-southeast-1 \
  > backup-route-table-$(date +%Y%m%d-%H%M%S).json

# Thêm route mới (ví dụ: on-prem CIDR mới qua TGW)
aws ec2 create-route \
  --route-table-id {{route-table-id}} \
  --destination-cidr-block {{10.xxx.0.0/xx}} \     # CIDR on-prem mới
  --transit-gateway-id tgw-074bbf044fada8981 \   # Luôn qua TGW cho on-prem
  --profile jp:network \
  --region ap-southeast-1

# Rollback: xóa route vừa thêm
# aws ec2 delete-route \
#   --route-table-id {{route-table-id}} \
#   --destination-cidr-block {{CIDR}} \
#   --profile jp:network --region ap-southeast-1

Ưu & Nhược điểm

Mô hình Network Account centralization của Japfa.

▾

Ưu điểm — Network Account Centralization

  • Kiểm soát tập trung — Toàn bộ VPC/Route/TGW trong 1 account, dễ audit và enforce policy
  • Workload account không tự tạo VPC — Không có VPC sprawl, không sai CIDR overlap
  • Airgap-DB subnet — RDS hoàn toàn cô lập, không có internet route từ thiết kế
  • 4 tier rõ ràng — Dễ hiểu: Public → Private-App → Airgap-DB → TGW-Attach
  • RAM Share efficient — 1 subnet phục vụ nhiều account, không duplicate VPC

Nhược điểm / Rủi ro

  • jp:network là SPOF quản lý — Mọi thay đổi network phải qua jp:network; nếu ai sửa nhầm sẽ ảnh hưởng toàn hạ tầng
  • CIDR không thể đổi — VPC CIDR đã chọn không thay đổi được; phải plan kỹ từ đầu
  • NAT GW là SPOF cho internet — 1 NAT GW per VPC; nếu NAT down thì EC2 private-app mất internet (không ảnh hưởng app vì chỉ cần inbound)
  • Không có IPv6 — Hiện chỉ dùng IPv4; nếu cần IPv6 phải thêm sau

Checklist Thêm / Sửa / Xóa

Checklist thay đổi VPC/Subnet/Route — ảnh hưởng toàn hệ thống.

▾
🔴
VPC/Subnet là foundation P1 — lỗi route table hoặc xóa subnet sẽ ngắt toàn bộ traffic production. LUÔN xác nhận 2 lần.

→Checklist: Thêm subnet mới

  • Plan CIDR không overlap — Verify CIDR mới không trùng với 10.83.0.0/18 và on-prem CIDRs
  • Chọn đúng AZ — Subnet AZ không thay đổi được sau khi tạo
  • Gắn Route Table đúng — Public: gắn RT có IGW; Private: NAT GW; Airgap: RT không có internet
  • Thêm vào RAM Share nếu cần share sang workload — Subnet mới mặc định không được shared
  • Tags đầy đủ — Name, Environment, Tier (public/private-app/airgap-db)
  • Import vào TF state — terraform import aws_subnet.<name> <subnet-id>

→Checklist: Sửa Route Table

  • Backup JSON route table — aws ec2 describe-route-tables --route-table-ids <id> > backup.json
  • Verify target gateway còn active — TGW, IGW, NAT GW phải ở trạng thái "available"
  • KHÔNG xóa route 0.0.0.0/0 của Public subnet — Mất IGW route = ALB mất internet
  • KHÔNG thêm internet route vào Airgap-DB — Vi phạm security design của RDS subnet
  • Test sau khi thêm route — Ping/curl từ instance trong subnet đó để verify route hoạt động

→Checklist: Xóa VPC/Subnet (cực kỳ nguy hiểm)

  • Verify không còn resource nào trong subnet — EC2, RDS, ALB, NAT GW, VPC Endpoint phải xóa/move trước
  • Xóa khỏi RAM Share trước — Revoke từ workload account trước khi xóa subnet
  • Backup toàn bộ VPC config — Export JSON tất cả: route tables, SG, NACL, subnet associations
  • Xác nhận 2 lần với 3T — VPC là foundation, xóa không thể undo

Best Practices & Tài liệu

Nguyên tắc thiết kế network Japfa và tài liệu tham khảo.

▾

→Best Practices

Security

  • Airgap DB subnet không có internet route — RDS không thể bị access từ internet, đây là security by design
  • NACL thêm lớp bảo vệ — SG kiểm soát instance level; NACL kiểm soát subnet level
  • Không bao giờ cho EC2 join Public subnet — EC2 webapp phải ở Private-App, không phải Public

Reliability

  • Mỗi tier có subnet ở 2 AZ — Đề phòng AZ failure. ALB cần 2 AZ tối thiểu
  • TGW-Attach cũng có 2 AZ — TGW ENI fail-safe qua 2 AZ

Operational Excellence

  • Tất cả trong jp:network TF module — Network không được tạo thủ công trong workload accounts
  • CIDR planning document — Ghi lại CIDR allocation để tránh conflict khi expand

→Gotchas & Pitfalls

PITFALL

jp:prod không thể sửa route table — Subnet được RAM share sang jp:prod nhưng route table owner là jp:network. Nếu cần thêm route: phải làm ở jp:network, không phải jp:prod.

PITFALL

VPC CIDR không thể thay đổi — Sau khi tạo VPC, CIDR cố định. Nếu cần mở rộng: chỉ có thể thêm secondary CIDR block (giới hạn 5 block/VPC).

GOTCHA

Airgap subnet không có NAT — EC2 trong airgap-db không thể download gì (kể cả AWS SSM agent update). Đây là intentional — chỉ RDS mới dùng subnet này.

TIP

Private-App subnet lớn (10.83.4.0/22 = 1024 IP) — Đủ chỗ cho hàng trăm EC2. Không lo hết IP trong tương lai gần.
