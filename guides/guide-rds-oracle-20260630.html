Cấu hình qua Console

Xem trạng thái và các thao tác thường gặp qua AWS Console.

▾
🔴
⚠️ ĐÂY LÀ PRODUCTION: Mọi thao tác Modify/Delete trên RDS đều cần xác nhận 2 lần. Snapshot trước khi sửa.

→Xem trạng thái RDS

  1. Đăng nhập Console jp:prod

    SSO → jp:prod (347486023795) → Console → Region: ap-southeast-1.

    Search: RDS → Databases → tìm jap-rds-prod-cdb-01.

    Status phải là "Available" — nếu thấy "Modifying" hoặc "Backing-up" là đang có thay đổi
  2. Kiểm tra các thông tin quan trọng
    Tab/FieldCần kiểm traGiá trị bình thường
    Summary → StatusTrạng thái RDS instanceAvailable
    Summary → Multi-AZCó standby khôngNo (Single-AZ — SPOF)
    Connectivity → EndpointConnection string cho ứng dụngjap-rds-prod-cdb-01.xxxxx.ap-southeast-1.rds.amazonaws.com:1521
    Maintenance & backups → Latest restore timePITR có hoạt động khôngTrong vài phút gần đây
    Logs & eventsCó lỗi gần đây khôngKhông có "Error" hoặc "Failover"
  3. Tạo snapshot thủ công (trước khi thay đổi)

    Databases → chọn jap-rds-prod-cdb-01 → Actions → Take snapshot.

    Đặt tên: jap-rds-prod-manual-backup-YYYYMMDD-HHMMSS

    Chờ status snapshot từ "Creating" → "Available" (5-10 phút) trước khi tiến hành thay đổi.

    Snapshot ID xuất hiện trong tab Snapshots — ghi lại để rollback nếu cần
  4. Modify RDS (nếu cần thay đổi cấu hình)

    Actions → Modify → thay đổi thông số cần → chọn khi apply:

    • Apply immediately: Thay đổi ngay lập tức — có thể gây downtime ngắn
    • Apply during next scheduled maintenance window: Áp dụng vào maintenance window định kỳ — an toàn hơn
    Với prod: LUÔN chọn "next maintenance window" trừ khi khẩn cấp

→Kết nối trực tiếp tới Oracle (qua Bastion)

  1. Mở SSM Session Manager tới Bastion

    EC2 Console → chọn bastion i-06c85f69071a2522d → Connect → Session Manager.

  2. Kết nối Oracle từ Bastion
    Cần có Oracle Instant Client trên bastion (Ubuntu 22.04). Kết nối port :1521 không qua internet.

→Lệnh đọc / kiểm tra (🟢 An toàn)

AWS CLI · Describe RDS🟢 READ ONLY
# Xem trạng thái RDS instance
aws rds describe-db-instances \
  --db-instance-identifier jap-rds-prod-cdb-01 \
  --profile jp:prod \
  --region ap-southeast-1 \
  --query 'DBInstances[0].{Status:DBInstanceStatus,Class:DBInstanceClass,Engine:EngineVersion,Multi:MultiAZ,Endpoint:Endpoint.Address}' \
  --output json

# Xem tất cả manual snapshots
aws rds describe-db-snapshots \
  --db-instance-identifier jap-rds-prod-cdb-01 \
  --snapshot-type manual \
  --query 'DBSnapshots[*].{ID:DBSnapshotIdentifier,Status:Status,Created:SnapshotCreateTime,Size:AllocatedStorage}' \
  --profile jp:prod --region ap-southeast-1 \
  --output table

# Xem PITR restore times (backup window)
aws rds describe-db-instances \
  --db-instance-identifier jap-rds-prod-cdb-01 \
  --query 'DBInstances[0].{BackupWindow:PreferredBackupWindow,MaintWindow:PreferredMaintenanceWindow,RetentionDays:BackupRetentionPeriod,LatestRestore:LatestRestorableTime}' \
  --profile jp:prod --region ap-southeast-1 --output json

# Xem CloudWatch metrics RDS (FreeStorageSpace)
aws cloudwatch get-metric-statistics \
  --namespace AWS/RDS \
  --metric-name FreeStorageSpace \
  --dimensions Name=DBInstanceIdentifier,Value=jap-rds-prod-cdb-01 \
  --start-time $(date -u -v-1H +%Y-%m-%dT%H:%M:%SZ) \
  --end-time $(date -u +%Y-%m-%dT%H:%M:%SZ) \
  --period 300 --statistics Average \
  --profile jp:prod --region ap-southeast-1 --output table

Ưu & Nhược điểm

Đánh giá RDS Oracle trong context Japfa.

▾

Ưu điểm

  • Fully managed — AWS lo patch OS, storage, backup tự động; không cần DBA quản lý OS
  • PITR 35 ngày — Khôi phục dữ liệu tới bất kỳ giây nào trong 35 ngày qua
  • CDB architecture — Gộp nhiều PDB (FARMIC, SDGL, UAT) vào 1 instance — tiết kiệm chi phí
  • Airgap subnet — Không có internet route — bảo mật cao, không thể tấn công từ ngoài
  • Oracle 19c SE2 — Đủ tính năng cho ERP/CRM quy mô Japfa Vietnam

Nhược điểm / Rủi ro

  • Single-AZ = SPOF — AZ-1a down → mất toàn bộ database. RTO thực tế 1-2 giờ
  • Chi phí cao — Oracle SE2 license + db.m5.xlarge ~$800-1000/tháng
  • SE2 giới hạn CPU — 2 socket tối đa; nếu workload tăng lên cần EE (gấp 3-4 lần giá)
  • Không scale horizontally — Read replica Oracle RDS không hỗ trợ trên SE2; chỉ scale up
  • UAT PDB trong prod RDS — Testing trực tiếp trong prod instance — có risk data leak UAT→prod

Checklist Thêm / Sửa / Xóa

Danh sách kiểm tra trước mỗi thao tác RDS Oracle.

▾
🔴
RDS Oracle là P1 — TUYỆT ĐỐI không thay đổi mà không có xác nhận 2 lần và snapshot backup.

→Checklist: Tạo mới RDS Oracle

  • Verify subnet airgap available — Subnet 10.83.16.0/24 và 10.83.17.0/24 đã được RAM share sang jp:prod
  • Chọn đúng engine: oracle-se2 — Không phải oracle-ee (đắt hơn x3-4)
  • License model: license-included — Không dùng BYOL trừ khi có Oracle license riêng
  • publicly_accessible = false — Tuyệt đối không expose RDS ra internet
  • Character set AL32UTF8 — Bắt buộc cho dữ liệu tiếng Việt
  • Backup retention 35 ngày — Tối đa để đảm bảo DR recovery
  • skip_final_snapshot = false — Bắt buộc snapshot khi xóa
  • prevent_destroy = true trong TF — Chống xóa nhầm bằng terraform destroy
  • Import vào Terraform state — terraform import aws_db_instance.oracle_cdb jap-rds-prod-cdb-01

→Checklist: Modify RDS (upgrade type, storage, Multi-AZ)

  • Tạo manual snapshot trước — Chờ "Available" trước khi tiến hành
  • Chọn maintenance window — Không apply-immediately trừ khi khẩn cấp
  • Thông báo users trước — BankSystem/CRM có thể bị downtime ngắn khi modify apply
  • Tăng storage: chỉ tăng, không giảm — RDS không cho phép giảm allocated storage
  • Multi-AZ: đánh giá chi phí trước — +~$500-700/tháng, cần approval management
  • Verify sau modify — Check status "Available", test kết nối từ webapp
  • Sync TF state nếu sửa thủ công — terraform plan để phát hiện drift

→Checklist: Xóa RDS (nguy hiểm cực cao)

🔴
KHÔNG BAO GIỜ xóa jap-rds-prod-cdb-01 mà không có sự đồng ý rõ ràng từ management. Đây là database duy nhất chứa toàn bộ dữ liệu production Japfa Vietnam.
  • Final snapshot bắt buộc — skip_final_snapshot phải = false
  • Export data ra S3 hoặc on-prem — Data pump export trước khi xóa
  • Xóa TF lifecycle prevent_destroy trước — Terraform sẽ block destroy nếu còn setting này
  • Xác nhận lần 2 với 3T + management — Đây là P1 resource, cần approval đặc biệt

Best Practices & Tài liệu

Nguyên tắc vàng và tài liệu tham khảo cho RDS Oracle Japfa.

▾

→Best Practices theo AWS Well-Architected

Security

  • Airgap subnet — không route internet — RDS chỉ accessible từ EC2 trong cùng VPC qua private IP
  • Security Group chặt — Chỉ mở :1521 từ SG của EC2 webapp, không CIDR 0.0.0.0/0
  • Encrypt at rest — KMS encryption cho storage đã bật
  • Rotate DB password định kỳ — Dùng Secrets Manager hoặc SSM Parameter Store

Reliability

  • PITR 35 ngày — Giữ retention tối đa để có thể recover data bị xóa nhầm
  • Manual snapshot trước mọi thay đổi — Không dựa hoàn toàn vào automated backup
  • Xem xét Multi-AZ — Single-AZ hiện tại là SPOF; nên đánh giá bật Multi-AZ cho prod
  • Test restore định kỳ — Verify snapshot có restore được mỗi quý một lần

Performance

  • Monitor FreeStorageSpace — Alert khi <20% storage còn trống (<100 GB / 500 GB)
  • Monitor DatabaseConnections — Oracle SE2 có connection limit; alert khi gần giới hạn
  • gp3 storage — Baseline 3000 IOPS miễn phí; có thể tune thêm IOPS nếu cần

Cost Optimization

  • Reserved Instance 1 năm — db.m5.xlarge Oracle SE2 ổn định → RI tiết kiệm ~35%
  • Xóa manual snapshot cũ không cần thiết — Mỗi 500GB snapshot = $47/tháng

→Gotchas & Pitfalls đặc thù Japfa

PITFALL

UAT PDB trong prod RDS — PDB "UAT" nằm trong cùng jap-rds-prod-cdb-01. Dev team test trên đây — có risk query nhầm vào PDB FARMIC/SDGL. Cần strict connection string và user separation.

PITFALL

Storage không giảm được — Một khi đã tăng allocated storage, không thể giảm xuống. Chỉ tăng khi chắc chắn cần thiết.

PITFALL

FeedProduction memory leak — App FeedProduction bị memory leak do Oracle DbContext không dispose + SAP retry loop. Log tại D:\OneDrive - JAPFA\Publish\FeedProduction\App_Data\Logs\. Gây nhiều connection tới RDS → theo dõi DatabaseConnections metric.

GOTCHA

BI Pipeline qua VPC Peering — Qlik/Snowflake Indonesia connect tới FARMIC PDB qua VPC Peering. Nếu RDS Security Group thay đổi, BI pipeline có thể bị gián đoạn mà không ảnh hưởng webapp.

TIP

PITR recovery — Nếu cần restore data bị xóa nhầm: dùng PITR restore tạo instance mới, export data từ đó, import vào prod. Không cần restore toàn bộ prod RDS.
