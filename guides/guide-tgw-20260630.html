Transit Gateway

Router trung tâm kết nối mọi VPC Japfa và tunnels VPN on-prem — hub-and-spoke topology, ASN 64512, owned by jp:network và shared sang tất cả workload accounts.

ID: tgw-074bbf044fada8981 ASN: 64512 Account: jp:network (787309524943) Region: ap-southeast-1 Criticality: P1
00 / 08

Tổng quan

Transit Gateway là "cloud router" — kết nối tất cả VPC Japfa và VPN on-prem vào 1 điểm duy nhất.

▾
Định nghĩa

AWS Transit Gateway (TGW) là regional router cho phép kết nối nhiều VPC và mạng on-prem vào 1 hub duy nhất. Japfa dùng mô hình hub-and-spoke: TGW là hub trung tâm, các VPC là spoke, VPN on-prem là external connections.

Cấu hình qua Console

Xem TGW, attachments, route tables và BGP routing qua Console.

▾
  1. Đăng nhập Console jp:network

    SSO → jp:network (787309524943) → Console → Region: ap-southeast-1.

    VPC → Transit Gateways → thấy TGW tgw-074bbf044fada8981.

  2. Xem TGW Attachments

    VPC → Transit Gateway Attachments → lọc theo Transit Gateway ID.

    Mỗi attachment có fields: State, Resource Type (vpc/vpn), Resource ID, Owner.

    State phải là "available" cho VPC attachment và "available" cho VPN attachment
  3. Xem TGW Route Tables

    VPC → Transit Gateway Route Tables → chọn route table → tab Routes.

    TGW có 1 default route table với:

    Destination CIDRNext HopTypeÝ nghĩa
    10.83.0.0/18VPC Attachment (prod)Static/PropagatedRoute tới Production VPC
    10.83.64.0/18VPC Attachment (poc)Static/PropagatedRoute tới POC VPC
    10.82.0.0/16VPN Attachment (LA)Propagated (BGP)On-prem Long An subnet
    0.0.0.0/0— không có ——Không có default route → packet unknown CIDR bị drop
  4. Xem BGP Status từ Site-to-Site VPN

    VPC → Site-to-Site VPN Connections → chọn connection → tab Tunnel details.

    Mỗi VPN có 2 tunnel (redundant). Xem: Status (UP/DOWN), Outside IP, BGP Status.

    BGP Status "UP" là điều kiện để route on-prem được propagate vào TGW route table

  --profile jp:network --region ap-southeast-1 \
  --output table

# Xem routes trong TGW route table (thay {{rt-id}} bằng TGW route table ID)
aws ec2 search-transit-gateway-routes \

  --filters "Name=state,Values=active" \

Cấu hình qua Terraform

TGW và attachments trong module terraform/network/ của Japfa.

▾
Terraform · tgw.tfHCL
# Transit Gateway definition
resource "aws_ec2_transit_gateway" "main" {
  description                     = "Japfa VN TGW — hub-and-spoke for VPC + VPN on-prem"
  amazon_side_asn                 = 64512           # AWS side BGP ASN — TIDAK bisa đổi sau khi tạo
  auto_accept_shared_attachments  = "enable"        # Auto accept VPC attachments từ shared accounts
  default_route_table_association = "enable"        # VPC attachment tự join default route table
  default_route_table_propagation = "enable"        # VPC/VPN routes tự propagate vào route table
  dns_support                     = "enable"        # DNS resolution across VPC
  vpn_ecmp_support                = "enable"        # Equal-cost multi-path cho VPN redundancy
  tags = { Name = "jp-network-tgw-aps1", ManagedBy = "terraform", Environment = "network" }
}

# VPC Attachment — Production VPC (cross-account từ jp:prod)
resource "aws_ec2_transit_gateway_vpc_attachment" "prod" {
  transit_gateway_id                              = aws_ec2_transit_gateway.main.id
  vpc_id                                          = aws_vpc.prod.id
  subnet_ids                                      = [aws_subnet.tgw_attach_1a.id, aws_subnet.tgw_attach_1b.id]
  transit_gateway_default_route_table_association = true
  transit_gateway_default_route_table_propagation = true
  dns_support                                     = "enable"
  tags = { Name = "jp-prod-tgw-att-vpc-prod" }
}

# RAM Share TGW sang jp:prod và jp:poc
resource "aws_ram_resource_association" "tgw" {
  resource_share_arn = aws_ram_resource_share.prod_subnets.arn
  resource_arn       = aws_ec2_transit_gateway.main.arn
}

Ưu & Nhược điểm

Transit Gateway vs VPN Direct vs VPC Peering.

▾

Ưu điểm

  • Hub-and-spoke scalable — Thêm VPC mới chỉ cần attach vào TGW, không cần mesh peering
  • VPN + VPC trong 1 router — On-prem và cloud đều qua TGW, 1 route table quản lý tất cả
  • BGP failover tự động — LA (primary) → HC (backup) tự chuyển khi LA fail
  • ECMP VPN — 2 tunnel per connection có thể dùng đồng thời (load balance)
  • Cross-account via RAM — 1 TGW phục vụ jp:prod, jp:poc, jp:shareservice

Nhược điểm / Rủi ro

  • Chi phí cao nhất trong network stack — ~$36/tháng TGW + $0.02/GB data processed
  • TGW là SPOF — Nếu TGW có vấn đề → toàn bộ on-prem mất kết nối AWS (dù AWS quản lý SLA 99.9%)
  • ASN không đổi được — amazon_side_asn = 64512 cố định từ lúc tạo
  • Complexity tăng — Debug routing phức tạp hơn VPC Peering đơn giản

Checklist Thêm / Sửa / Xóa

TGW là P1 — mọi thay đổi cần backup + xác nhận 2 lần.

▾
🔴
Transit Gateway là backbone routing P1. Mọi thay đổi có thể ảnh hưởng toàn bộ kết nối on-prem ↔ AWS. Backup JSON trước, xác nhận 2 lần.

→Checklist: Attach VPC mới vào TGW

  • Tạo TGW-Attach subnet — Mỗi VPC cần 2 subnet riêng cho TGW ENI (ít nhất /28 mỗi AZ)
  • Verify CIDR không overlap — CIDR VPC mới không được trùng với bất kỳ CIDR nào đang có trong TGW route table
  • RAM Share TGW cho account mới — Nếu VPC thuộc account khác, phải share TGW trước
  • Tạo VPC attachment — Chỉ định subnet TGW-Attach, enable propagation
  • Cập nhật VPC Route Table — Thêm route 10.82.0.0/16 → TGW trong Route Table của VPC mới để traffic on-prem đi được
  • Test connectivity — Ping từ EC2 trong VPC mới tới on-prem IP

→Checklist: Sửa TGW Route Table

  • Xác định route table ID — aws ec2 describe-transit-gateway-route-tables
  • Backup routes hiện tại — aws ec2 search-transit-gateway-routes ... > backup.json
  • Verify attachment ID đích — Đảm bảo route tới đúng VPN attachment, không phải VPC attachment nhầm
  • Không xóa propagated routes — Chỉ quản lý static routes; propagated routes là do BGP tự cập nhật
  • Test sau thay đổi — Traceroute từ EC2 tới on-prem IP verify route đi đúng path

Best Practices & Tài liệu

Nguyên tắc vận hành TGW Japfa và tài liệu tham khảo.

▾

→Best Practices

Security

  • Chỉ jp:network manage TGW — Workload accounts (jp:prod, jp:poc) không có quyền sửa TGW, chỉ attach VPC
  • RAM Share với allow_external_principals = false — Chỉ share trong cùng AWS Organization Japfa
  • Không có default route 0.0.0.0/0 trong TGW — Packet tới CIDR không biết bị drop, không ra internet

Reliability

  • Luôn có 2 tunnel per VPN connection — AWS tự load balance / failover giữa 2 tunnels
  • HC làm backup qua BGP AS-prepend — Tự động failover mà không cần manual intervention
  • ECMP enabled — Cho phép dùng cả 2 tunnel của same VPN cùng lúc

Monitoring

  • Monitor BytesIn/BytesOut per attachment — Phát hiện data spike bất thường
  • VPN tunnel status alarm — Alarm khi tunnel DOWN (đã có vpn-autoheal lambda)

→Gotchas & Pitfalls

PITFALL

TGW Attachment phải ở TGW-Attach subnet riêng — Không dùng Private-App hay Public subnet cho TGW ENI. TGW cần /28 nhỏ trong subnet riêng (đã tách ra: 10.83.20.0/28 per AZ).

PITFALL

ASN 64512 cố định — amazon_side_asn không thể thay đổi sau khi TGW được tạo. FortiGate phải dùng ASN khác (65000). Không để 2 bên dùng cùng ASN.

GOTCHA

Propagated route vs Static route — BGP route từ FortiGate là "propagated" — tự cập nhật theo BGP state. Static route là cố định. KHÔNG xóa propagated route — nó tự quản lý bởi BGP.

TIP

VPN auto-heal lambda — Japfa đã có Lambda tự reset VPN tunnel khi DOWN. Guide riêng: guide-vpn-20260630.html.
