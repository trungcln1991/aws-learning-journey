AWS CLI · Describe Instances🟢 READ ONLY

  --profile jp:prod \
  --region ap-southeast-1 \
  --query 'Reservations[*].Instances[*].{ID:InstanceId,Name:Tags[?Key==`Name`]|[0].Value,State:State.Name,Type:InstanceType,IP:PrivateIpAddress}' \
  --output table

# Chi tiết 1 instance cụ thể (webapp)
aws ec2 describe-instances \
  --instance-ids i-0cb8bae3be071dc0e \
  --profile jp:prod \
  --region ap-southeast-1 \
  --output json

# Kiểm tra IMDSv2 status
aws ec2 describe-instances \
  --instance-ids i-0cb8bae3be071dc0e \
  --query 'Reservations[0].Instances[0].MetadataOptions' \
  --profile jp:prod --region ap-southeast-1 \
  --output json

# Xem CloudWatch metric CPUUtilization 1 giờ gần nhất
aws cloudwatch get-metric-statistics \
  --namespace AWS/EC2 \
  --metric-name CPUUtilization \
  --dimensions Name=InstanceId,Value=i-0cb8bae3be071dc0e \
  --start-time $(date -u -v-1H +%Y-%m-%dT%H:%M:%SZ) \
  --end-time $(date -u +%Y-%m-%dT%H:%M:%SZ) \
  --period 300 --statistics Average \
  --profile jp:prod --region ap-southeast-1 \
  --output table

resource "aws_instance" "webapp" {

  # ── Bắt buộc ──────────────────────────────────
  ami           = "ami-0xxxxxxxxxxxxxxxx"  # [BẮT BUỘC] AMI Windows Server 2016 ap-southeast-1
  instance_type = "m5.2xlarge"            # [BẮT BUỘC] 8 vCPU, 32GB RAM cho IIS webapp
  subnet_id     = data.aws_subnet.private_app_1a.id # RAM share từ jp:network

  # ── Network ─────────────────────────────────
  vpc_security_group_ids = [aws_security_group.webapp.id]
  associate_public_ip_address = false     # [QUAN TRỌNG] Không bao giờ public IP

  # ── IAM ─────────────────────────────────────
  iam_instance_profile = "jp-ec2-ssm-role" # SSM + CloudWatch + S3 access

  # ── IMDSv2 (bắt buộc) ───────────────────────
  metadata_options {
    http_endpoint               = "enabled"
    http_tokens                 = "required"  # IMDSv2 enforce
    http_put_response_hop_limit = 1
  }

  # ── Storage ─────────────────────────────────
  root_block_device {
    volume_type           = "gp3"          # gp3 rẻ hơn gp2, tốt hơn cho IIS
    volume_size           = 100            # GB cho ổ C:
    delete_on_termination = true
    encrypted             = true
  }

  # ── Tags bắt buộc ─────────────────────────
  tags = {
    Name        = "jp-prod-ec2-pv-aps11-webapp"
    Environment = "production"
    ManagedBy   = "terraform"
    Owner       = "trungcln"
    Backup      = "jp-prod-backup-plan-ec2"
  }

  # ── Lifecycle ───────────────────────────────
  lifecycle {
    # Japfa ignore AMI changes — update AMI phải làm có kế hoạch, không tự replace
    ignore_changes = [ami]
  }
}

# Data source — subnet từ jp:network (RAM share)
data "aws_subnet" "private_app_1a" {
  id = "subnet-xxxxxxxxxxxxxxxxx"  # Private-App AZ-1a 10.83.4.0/22 (jp:network → RAM → jp:prod)
}

Ưu & Nhược điểm

Đánh giá EC2 trong context Japfa — so với giải pháp thay thế.

▾

Ưu điểm

  • Full control OS — Cài IIS, Oracle Client, custom libraries không bị giới hạn như container
  • Lift-and-shift dễ dàng — MGN migrate từ VMware on-prem lên, không cần refactor app
  • Windows workload native — IIS + .NET + RDP — không cần containerize
  • SSM Session Manager — Quản trị an toàn không cần bastion public
  • AMI snapshot nhanh — Backup toàn bộ OS + data, rollback <1 giờ

Nhược điểm / Rủi ro

  • Single AZ — SPOF — Webapp và bastion cùng AZ-1a, không có failover tự động nếu AZ down
  • Disk ổ D: 87% — Nguy cơ đầy disk, chưa có alert tự động cho disk usage
  • Chi phí cao hơn serverless — m5.2xlarge chạy 24/7 dù traffic thấp ban đêm
  • OS patch manual coordination — Auto-patch có risk downtime, cần maintenance window
  • No Auto Scaling — Không scale ngang nếu traffic đột biến

→So sánh với giải pháp thay thế

Giải phápTình huống phù hợpJapfa dùng?Lý do lựa chọn
EC2 (đang dùng) Windows workload, full control OS, lift-and-shift ✓ Đang dùng IIS/.NET app migrate từ VMware, không cần refactor
ECS/EKS Container Microservices, modern app, cần scale ngang — Không dùng App cũ .NET Framework không containerize dễ dàng
AWS Elastic Beanstalk .NET app managed platform — Không dùng Ít control hơn EC2, khó customize IIS settings
Lambda + API GW Event-driven, stateless functions — Không dùng (SCP chặn Lambda) SCP Org chặn Lambda ở jp:prod; app monolithic không phù hợp

Checklist Thêm / Sửa / Xóa

Danh sách kiểm tra trước mỗi thao tác EC2 quan trọng tại Japfa.

▾
⚠
Nhắc nhở: Mọi thao tác 🟡🔴 cần xác nhận 2 lần với 3T. Checklist này không thay thế xác nhận đó.

→Checklist: Tạo mới EC2

  • SSO login đúng account — aws sso login --profile jp:prod, verify aws sts get-caller-identity
  • Xác nhận subnet RAM share available — Subnet private-app đã được share từ jp:network chưa
  • Chọn AMI đúng region ap-southeast-1 — Không dùng AMI của region khác
  • Không gắn Public IP — associate_public_ip_address = false
  • Gắn IAM Profile — jp-ec2-ssm-role (bắt buộc cho SSM access)
  • Tags bắt buộc — Name, Environment, ManagedBy=terraform, Owner=trungcln
  • Set IMDSv2 ngay sau launch — HttpTokens=required (Security Hub sẽ flag nếu bỏ qua)
  • Gắn Security Group phù hợp — Chỉ mở port cần thiết, không 0.0.0.0/0 cho admin ports
  • Import vào Terraform state — terraform import aws_instance.<name> <instance-id>
  • Verify SSM connectivity — Mở Session Manager từ console để test
  • Cập nhật infra-overview.md — Ghi instance ID + type + vai trò mới

→Checklist: Chỉnh sửa EC2 (type, storage, SG)

  • Backup AMI trước — aws ec2 create-image --no-reboot + chờ available
  • Thông báo maintenance window — Sửa type/stop cần báo trước cho user (BankSystem/CRM)
  • Stop instance nếu đổi type — Instance type chỉ đổi được khi stopped
  • Sửa SG: không xóa rule cũ trước khi test rule mới — Add trước, verify, rồi xóa rule cũ
  • Rollback plan rõ ràng — Biết lệnh restore AMI trước khi apply
  • Verify sau sửa — Check ALB target health, test app BankSystem/CRM
  • Sync TF state — Nếu sửa thủ công: terraform plan để phát hiện drift
  • Update infra-overview.md — Ghi thay đổi type/config mới

→Checklist: Xóa / Terminate EC2

🔴
NGUY HIỂM: Terminate instance không thể phục hồi (ngoại trừ từ AMI backup). Phải có sự đồng ý rõ ràng từ 3T.
  • Backup AMI đầy đủ — Tạo AMI, chờ available, ghi lại AMI ID
  • Kiểm tra dependents — Instance còn trong ALB target group không? Đã detach chưa?
  • Detach khỏi ALB Target Group trước — Tránh health check fail lan rộng
  • Disable CloudWatch alarm liên quan — Tránh false alarm sau khi terminate
  • Xác nhận lần 2 với 3T — Đọc to Instance ID cần xóa: i-0xxxxxxxxxxxxxxxxx
  • Remove khỏi TF state — terraform state rm aws_instance.<name>
  • Verify sau terminate — Confirm billing đã dừng trong Cost Explorer ngày hôm sau
  • Update infra-overview.md — Xóa entry instance đã terminate

Best Practices & Tài liệu

Nguyên tắc vàng và tài liệu tham khảo cho EC2 Japfa.

▾

→Best Practices theo AWS Well-Architected

Security

  • IMDSv2 bắt buộc — HttpTokens=required trên tất cả instances (đã done 22/06/2026)
  • Không mở SSH/RDP ra internet — Dùng SSM Session Manager qua VPC Endpoint
  • IAM Role thay vì access key — Instance dùng role jp-ec2-ssm-role, không nhúng credentials
  • Encrypt EBS volumes — Bật encryption cho tất cả EBS (at-rest encryption)
  • Security Group least-privilege — Chỉ mở port cần thiết, source cụ thể (SG ID hoặc CIDR on-prem)

Reliability

  • AMI backup trước mọi thay đổi — Auto-patch đã có bước backup AMI tự động
  • Health check ALB — 3 target groups (banksystem/crm/feedprod) đều được check sau patch
  • Monitor disk usage ổ D: — Hiện 87%, cần alert khi >90% để tránh IIS crash
  • Power schedule chỉ cho dev — Không bao giờ schedule stop cho webapp prod

Performance

  • EBS gp3 cho storage — gp3 rẻ hơn gp2 và có thể tune IOPS/throughput độc lập
  • Monitor CPU trước khi upgrade — Verify CPU utilization trung bình trước khi quyết định scale up

Cost Optimization

  • Power schedule cho dev — office-hours tag: start 7h/stop 23h → tiết kiệm ~65% dev cost
  • Reserved Instance nếu chạy >1 năm — Webapp prod ổn định, RI 1 năm tiết kiệm 30-40%

Operational Excellence

  • Tất cả instances phải trong TF state — Không tạo thủ công mà không import
  • Auto-patch theo schedule — Webapp: CN tuần 3 02:00; Dev: thứ 7 06:30; Bastion: thứ 7 01:00
  • Tags đầy đủ — Name, Environment, ManagedBy, Owner, Backup

→Gotchas & Pitfalls đặc thù Japfa

PITFALL

PlatformDetails báo nhầm OS — API EC2 PlatformDetails webapp báo "Linux/UNIX" nhưng thực tế Windows Server 2016. Luôn verify OS qua SSM Fleet Manager hoặc Inspector, không tin API này khi troubleshoot.

PITFALL

Disk ổ D: webapp 87% — Hiện 104/120 GB. Nếu đầy 100%, IIS log và session files bị lỗi → app crash. Cần alert CloudWatch cho disk usage >90%.

PITFALL

SCP Org chặn Lambda trong jp:prod — Không thể tạo Lambda function trong account này. Auto-patch dùng SSM Automation thay vì Lambda.

GOTCHA

Subnet thuộc jp:network — không xóa được từ jp:prod — Subnet và VPC do jp:network sở hữu qua RAM Share. jp:prod chỉ launch instance, không quản lý network layer.

GOTCHA

Windows TLS 1.2 — Windows 2016 mặc định dùng TLS 1.0 cho PowerShell Invoke-WebRequest → Teams webhook bị lỗi. Fix: [Net.ServicePointManager]::SecurityProtocol = [Net.SecurityProtocolType]::Tls12 trong SSM scripts.

TIP

AMI tag ManagedBy=ssm-autopatch — Auto-patch luôn tag AMI với key này. Khi rollback, filter AMI theo tag này để tìm backup gần nhất của từng instance.

TIP

Dev instance: verify power-schedule — Nếu dev không start sau 7h, check EventBridge Scheduler jp-prod-sch-dev-autopatch và SSM document JapfaDev-WindowsAutoPatch.
