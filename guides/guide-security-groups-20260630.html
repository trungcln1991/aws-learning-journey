Tổng quan

SG chain 3 lớp bảo vệ production Japfa — mỗi SG chỉ allow từ SG trước đó.

▾
🔴

⚠️ ĐÂY LÀ PRODUCTION — Sửa SG sai có thể lock out toàn bộ users (xóa allow rule) hoặc expose RDS/EC2 ra internet (thêm 0.0.0.0/0). Backup JSON + xác nhận 2 lần.

Nguyên tắc Japfa

Security Groups Japfa dùng SG-to-SG reference: EC2 chỉ nhận traffic từ SG của ALB, không phải từ internet. RDS chỉ nhận từ SG của EC2. Không bao giờ dùng 0.0.0.0/0 cho inbound (trừ Public ALB với :443/:80).

Cấu hình qua Console

Xem và quản lý Security Groups qua Console jp:prod.

▾
  1. Đăng nhập Console jp:prod — xem Security Groups

    SSO → jp:prod (347486023795) → EC2 → Security Groups → filter theo VPC production.

    Xem danh sách SG — tên convention: jp-prod-sg-*.

  2. Audit Inbound Rules — verify không có 0.0.0.0/0 cho internal ports

    Click vào từng SG → tab Inbound rules.

    Kiểm tra:

    SGPortSource phải làCần lo nếu thấy
    SG-EC2-Webapp:62375/:62404/...SG-ALB-Public ID0.0.0.0/0 = lỗ hổng bảo mật
    SG-RDS:1521SG-EC2-Webapp ID0.0.0.0/0 hoặc bất kỳ CIDR nào khác = CRITICAL
    SG-ALB-Public:443/:800.0.0.0/0 (OK)Thiếu :80 = user không redirect được về HTTPS
  3. Xem Security Group associations (SG đang gắn vào resource nào)

    EC2 → Security Groups → chọn SG → tab Associated resources (hoặc dùng CLI).

    Verify: SG-RDS chỉ gắn vào RDS instance. SG-EC2-Webapp chỉ gắn vào EC2 webapp và bastion
  4. Export JSON backup trước khi sửa

    Dùng CLI để export (xem Section S2). Console không có export built-in.

    Backup bắt buộc trước MỌI thay đổi SG production.

Cấu hình qua CLI

AWS CLI để đọc, audit và sửa Security Groups.

▾

→Lệnh đọc và audit (🟢 An toàn)

AWS CLI · Describe Security Groups🟢 READ ONLY · jp:prod
# Liệt kê tất cả SG trong Production VPC
aws ec2 describe-security-groups \
  --filters "Name=vpc-id,Values={{vpc-id-prod}}" \
  --profile jp:prod --region ap-southeast-1 \
  --query 'SecurityGroups[*].{ID:GroupId,Name:GroupName,Desc:Description}' \
  --output table

# Xem chi tiết inbound/outbound rules của 1 SG
aws ec2 describe-security-groups \
  --group-ids {{sg-id}} \
  --profile jp:prod --region ap-southeast-1 \
  --output json

# BACKUP toàn bộ SG trước khi thay đổi
aws ec2 describe-security-groups \
  --filters "Name=vpc-id,Values={{vpc-id-prod}}" \
  --profile jp:prod --region ap-southeast-1 \
  > backup-security-groups-$(date +%Y%m%d-%H%M%S).json

# Audit: tìm SG nào có inbound 0.0.0.0/0 (ngoài Public ALB)
aws ec2 describe-security-groups \
  --filters "Name=vpc-id,Values={{vpc-id-prod}}" \
  --profile jp:prod --region ap-southeast-1 \
  --query 'SecurityGroups[?IpPermissions[?IpRanges[?CidrIp==`0.0.0.0/0`]]].{Name:GroupName,ID:GroupId}' \
  --output table

# Xem SG nào đang gắn vào EC2 instances
aws ec2 describe-instances \
  --profile jp:prod --region ap-southeast-1 \
  --query 'Reservations[*].Instances[*].{ID:InstanceId,SGs:SecurityGroups[*].GroupId}' \
  --output table

→Thêm / Xóa inbound rule (🔴 Production)

🔴
⚠️ ĐÂY LÀ PRODUCTION — backup JSON trước + xác nhận 2 lần. Xóa rule inbound sai = lock out users / apps. Thêm rule sai = security gap.
AWS CLI · Modify Security Group Rules🔴 WRITE · jp:prod
# Thêm inbound rule — SG reference (ưu tiên) thay vì CIDR
aws ec2 authorize-security-group-ingress \
  --group-id {{sg-id-target}} \
  --ip-permissions '[{
    "IpProtocol":"tcp",
    "FromPort":62375,
    "ToPort":62375,
    "UserIdGroupPairs":[{"GroupId":"{{sg-id-source}}"}]
  }]' \
  --profile jp:prod --region ap-southeast-1

# Thêm inbound CIDR (on-prem) — chỉ dùng khi không thể dùng SG reference
aws ec2 authorize-security-group-ingress \
  --group-id {{sg-id}} \
  --protocol tcp --port 62375 \
  --cidr 10.82.0.0/16 \      # On-prem Long An CIDR
  --profile jp:prod --region ap-southeast-1

# Xóa inbound rule (rollback) — cần biết chính xác rule đang có
# aws ec2 revoke-security-group-ingress \
#   --group-id {{sg-id}} \
#   --protocol tcp --port 62375 --cidr {{CIDR}} \
#   --profile jp:prod --region ap-southeast-1

Cấu hình qua Terraform

Security Groups trong TF jp:prod — SG reference pattern.

▾
Terraform · security_groups.tfHCL
# SG Public ALB — nhận :443/:80 từ internet
resource "aws_security_group" "alb_public" {
  name        = "jp-prod-sg-alb-public"
  description = "Public ALB — HTTPS/HTTP from internet"
  vpc_id      = data.aws_vpc.prod.id

  ingress {                        # HTTPS từ internet — Public ALB cần 0.0.0.0/0
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
  ingress {                        # HTTP → redirect về HTTPS
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
  egress {
    from_port       = 0
    to_port         = 0
    protocol        = "-1"
    security_groups = [aws_security_group.ec2_webapp.id]  # Outbound chỉ tới EC2 SG
  }
}

# SG EC2 Webapp — nhận từ SG ALB (không nhận từ internet trực tiếp)
resource "aws_security_group" "ec2_webapp" {
  name        = "jp-prod-sg-ec2-webapp"
  description = "EC2 Webapp IIS — allow only from ALB SGs and on-prem"
  vpc_id      = data.aws_vpc.prod.id

  ingress {                        # BankSystem
    from_port       = 62375
    to_port         = 62375
    protocol        = "tcp"
    security_groups = [aws_security_group.alb_public.id]  # SG reference — best practice
  }
  ingress {                        # CRM
    from_port       = 62404; to_port = 62404; protocol = "tcp"
    security_groups = [aws_security_group.alb_public.id]
  }
  ingress {                        # E-Approval FE
    from_port       = 62408; to_port = 62408; protocol = "tcp"
    security_groups = [aws_security_group.alb_public.id]
  }
  ingress {                        # E-Approval Backend (từ Internal ALB)
    from_port       = 62167; to_port = 62167; protocol = "tcp"
    security_groups = [aws_security_group.alb_internal.id]
  }
  ingress {                        # On-prem users qua VPN/TGW (Long An + Hương Canh)
    from_port   = 62375; to_port = 62408; protocol = "tcp"
    cidr_blocks = ["10.82.0.0/16"]    # On-prem CIDR — cần CIDR vì on-prem không có SG
  }
  egress {                         # Outbound: Oracle DB và AWS APIs/SSM
    from_port       = 1521; to_port = 1521; protocol = "tcp"
    security_groups = [aws_security_group.rds_oracle.id]
  }
  egress {                         # :443 outbound cho SSM, patch, AWS APIs
    from_port   = 443; to_port = 443; protocol = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

# SG RDS Oracle — CHỈ allow :1521 từ SG-EC2-Webapp
resource "aws_security_group" "rds_oracle" {
  name        = "jp-prod-sg-rds-oracle"
  description = "RDS Oracle — allow Oracle port ONLY from EC2 Webapp SG"
  vpc_id      = data.aws_vpc.prod.id

  ingress {
    from_port       = 1521
    to_port         = 1521
    protocol        = "tcp"
    security_groups = [aws_security_group.ec2_webapp.id, aws_security_group.bastion.id]
  }
  # KHÔNG có ingress rule nào khác — RDS không accessible từ internet
}

Ưu & Nhược điểm

SG Reference vs CIDR-based vs NACL.

▾

Ưu điểm — SG Reference Pattern

  • Tự động theo scale — Khi EC2 IP thay đổi, SG rule vẫn đúng vì reference là SG ID
  • Stateful — Không cần viết outbound rules cho established connections (khác NACL)
  • Least privilege — Mỗi SG chỉ allow từ source tối thiểu cần thiết
  • Audit dễ — Nhìn rule SG reference hiểu ngay "từ resource nào"
  • Free — SG không tính tiền per rule hay per SG

Nhược điểm / Rủi ro

  • Không ngăn được intra-SG traffic — Mặc định SG cho phép instance trong cùng SG communicate. Cần thêm SG self-reference rule nếu muốn block
  • Circular dependency trong TF — SG A tham chiếu SG B và ngược lại → TF deadlock. Phải dùng aws_security_group_rule riêng
  • SG limits — Tối đa 60 inbound rules per SG, 5 SG per ENI (EC2)
  • Không block by default — Stateful SG: nếu không có rule → traffic bị drop (implicit deny)

→So sánh SG vs NACL

Checklist Thêm / Sửa / Xóa

Checklist Security Group changes trên Production.

▾

→Checklist: Thêm inbound rule (port/service mới)

  • Backup JSON hiện tại — aws ec2 describe-security-groups --group-ids {{sg-id}} > backup.json
  • Dùng SG reference thay vì CIDR — Nếu source là AWS resource → dùng SG ID. Chỉ dùng CIDR cho on-prem (10.82.0.0/16)
  • Verify port đang chạy trên EC2 — netstat -an | findstr :62xxx trên EC2 để confirm IIS đang listen port đó
  • Không thêm 0.0.0.0/0 cho internal ports — Chỉ Public ALB mới có :443/:80 từ 0.0.0.0/0
  • Test connectivity sau khi thêm — curl/telnet từ source tới target port
  • Update TF state — Import rule mới hoặc update TF file để TF state khớp

→Checklist: Xóa inbound rule

  • Verify không còn service nào cần rule đó — Check còn app nào dùng port/source đó không
  • Backup JSON trước — Lưu full SG config để có thể rollback chính xác
  • Xác nhận 2 lần với 3T — Xóa rule inbound = block traffic; không thể undo tức thì nếu đã xóa
  • Test sau khi xóa — Verify traffic từ source cũ không còn reach target

Best Practices & Tài liệu

Nguyên tắc Security Group Japfa.

▾

→Best Practices

Security

  • SG Reference > Prefix List > CIDR > 0.0.0.0/0 — Thứ tự ưu tiên khi chọn source/destination
  • Không để SSH :22 hay RDP :3389 mở từ 0.0.0.0/0 — Dùng SSM Session Manager thay vì SSH/RDP trực tiếp
  • RDS SG inbound chỉ từ EC2 SG — Không bao giờ allow :1521 từ 0.0.0.0/0 hay bất kỳ CIDR nào ngoài EC2 SG
  • Outbound restrictive khi có thể — EC2 outbound chỉ :1521 tới RDS SG và :443 ra internet (SSM, patch)
  • Audit định kỳ — Tìm SG có 0.0.0.0/0 inbound (trừ Public ALB)

Naming Convention

  • jp-prod-sg-{resource}-{purpose} — Ví dụ: jp-prod-sg-ec2-webapp, jp-prod-sg-rds-oracle
  • Description bắt buộc mô tả purpose và allowed sources

→Gotchas

PITFALL

Circular dependency TF với SG reference — Nếu SG A egress reference SG B, và SG B egress reference SG A → TF không tạo được. Fix: dùng aws_security_group_rule resource riêng thay vì inline ingress/egress block.

PITFALL

Xóa inbound rule không rollback ngay được — Sau khi revoke rule, connection hiện tại bị drop. Phải re-add rule và user reconnect. Không có "undo" tức thì.

GOTCHA

On-prem cần CIDR vì không có SG — On-prem qua VPN không có AWS SG ID → phải dùng CIDR 10.82.0.0/16. Đây là ngoại lệ hợp lý duy nhất cho CIDR inbound EC2.

TIP

Kiểm tra inbound 0.0.0.0/0 định kỳ — Chạy aws ec2 describe-security-groups --query '...[?IpRanges[?CidrIp==`0.0.0.0/0`]]' để phát hiện SG có rule quá rộng.
